# Roadmap: Writ Compiler — CST-to-AST Lowering Pipeline (v1.0)

## Overview

This milestone builds the first real code in `writ-compiler`: a multi-pass pipeline that transforms the full-fidelity CST produced by `writ-parser` into a simplified AST suitable for downstream semantic analysis. The pipeline desugars all Writ higher-level constructs — dialogue blocks, entities, operator overloads, optional sugar, formattable strings, and compound assignments — into their spec-defined primitives. The delivery order follows the hard dependency chain: AST types and pipeline infrastructure come first because every pass emits into them; foundational expression desugarings come second because structural passes invoke them as helpers; complex stateful passes (dialogue, entity) come last. Integration and snapshot testing close the milestone by validating the assembled pipeline end-to-end.

## Phase Details

### Phase 1: AST Foundation
**Goal**: The AST type hierarchy and pipeline infrastructure exist — all passes have a target type to emit into, all errors have a type to emit into, and the public API compiles
**Depends on**: Nothing (first phase)
**Requirements**: R1, R2, R14
**Success Criteria** (what must be TRUE):
  1. `AstExpr`, `AstStmt`, `AstDecl`, `AstType` are defined with no CST sugar variants and no `'src` lifetime
  2. Every AST node constructor requires a `span: SimpleSpan` argument — no exceptions and no `SimpleSpan::new(0, 0)` tombstones
  3. `LoweringContext` compiles with its error accumulator, speaker stack, and localization key counter
  4. `lower(items: Vec<Spanned<Item<'_>>>) -> (Ast, Vec<LoweringError>)` compiles as a stub that returns an empty AST
  5. `LoweringError` carries a source `SimpleSpan` and a descriptive message, powered by `thiserror`
**Plans:** 2 plans
- [x] 01-01-PLAN.md — Restructure crate and define AST type hierarchy (AstExpr, AstStmt, AstDecl, AstType)
- [x] 01-02-PLAN.md — Pipeline infrastructure (LoweringError, LoweringContext, lower() stub)

### Phase 2: Foundational Expression Lowering
**Goal**: Optional sugar, formattable strings, and compound assignments are lowered — the shared helpers that every structural pass invokes when it encounters type positions and expression positions
**Depends on**: Phase 1
**Requirements**: R3, R4, R5
**Success Criteria** (what must be TRUE):
  1. `T?` in any type position (param, return, field, local) lowers to `Option<T>` in the AST; `null` lowers to `Option::None`
  2. `$"Hello {name}!"` lowers to a string concatenation chain with `.into<string>()` calls; escaped `{{` and `}}` produce literal braces
  3. All five compound assignment operators (`+=`, `-=`, `*=`, `/=`, `%=`) lower to their expanded `a = a op b` form
  4. Each of these three lowerings has `insta` snapshot tests covering its core transformations
**Plans:** 2/2 plans complete
- [x] 02-01-PLAN.md — Core lowering functions (lower_type, lower_expr, lower_fmt_string, lower_stmt, lower_fn) and pipeline wiring
- [x] 02-02-PLAN.md — Insta snapshot tests for R3 (optional sugar), R4 (formattable strings), R5 (compound assignments)

### Phase 3: Operator and Concurrency Lowering
**Goal**: Expression-level lowering coverage is complete — operator overloads desugar to contract method calls and concurrency primitives survive lowering as first-class AST nodes
**Depends on**: Phase 2
**Requirements**: R6, R7
**Success Criteria** (what must be TRUE):
  1. `operator +(other: T) -> T { ... }` inside an `impl` block lowers to an `impl Add<T, T> for Self` with the body intact
  2. Derived operators are auto-generated: `!=` from `Eq`, `>` from `Ord`, `<=` and `>=` from `Eq` + `Ord`
  3. `spawn`, `join`, `cancel`, `defer`, and `detached` each map to their corresponding `AstExpr` or `AstStmt` variant with span preserved and no semantic transformation
  4. Snapshot tests cover binary operator desugaring, unary operators, index operators, and concurrency pass-through
**Plans:** 2/2 plans complete
- [x] 03-01-PLAN.md — Implement lower/operator.rs with operator-to-contract desugaring and wire into lower/mod.rs
- [x] 03-02-PLAN.md — Snapshot tests for R6 operator lowering and R7 concurrency pass-through

### Phase 4: Dialogue Lowering and Localization
**Goal**: `dlg` blocks are fully lowered to `fn` declarations with correct speaker resolution, say/choice/transition semantics, localization keys, and all dialogue-specific validation errors
**Depends on**: Phase 3
**Requirements**: R8, R9, R10, R11
**Success Criteria** (what must be TRUE):
  1. `@Speaker text` lines lower to `say_localized(speaker_ref, key, "text")` calls; `@Speaker` standalone lines set the active speaker for all subsequent lines in the current scope without leaking into sibling `$ choice` branches
  2. Speaker resolution correctly handles all three tiers: (1) local parameters/variables referenced directly, (2) `[Singleton]` entities via `Entity.getOrCreate<T>()`, (3) unknown speakers produce a `LoweringError` with the originating span while lowering continues for the rest of the block
  3. `-> target` and `-> target(args)` lower to `return target()` / `return target(args)` — always wrapped in `return`; `->` appearing before the end of its block produces a `LoweringError`
  4. Auto-generated FNV-1a localization keys incorporate block name and line index — two dialogue blocks with identical text produce distinct keys; manual `#key` overrides replace auto-keys; duplicate `#key` values within a block produce a `LoweringError`
  5. Snapshot tests cover: three-tier speaker resolution, `$ choice` branch speaker scoping, `->` transition lowering, `#key` override and collision, `{expr}` interpolation in dialogue text
**Plans**: 2 plans
- [x] 04-01-PLAN.md — Core dialogue lowering: lower_dialogue(), all 8 DlgLine variants, FNV-1a keys, speaker resolution, choice scoping, transition validation
- [x] 04-02-PLAN.md — Snapshot tests for R8-R11 dialogue lowering and localization

### Phase 5: Entity Lowering
**Goal**: `entity` declarations are fully lowered to a struct, per-component `ComponentAccess<T>` impls, and lifecycle hook registrations
**Depends on**: Phase 2
**Requirements**: R12, R13
**Success Criteria** (what must be TRUE):
  1. `entity Name { ... }` lowers to `struct Name` with separate fields for entity properties and per-component storage, plus one `impl ComponentAccess<T>` per `use` clause
  2. `use Health { current: 80, max: 80 }` correctly merges the entity-level field overrides with the component's own defaults — fields not overridden in the `use` clause use the component's defaults
  3. Two components with a method of the same name produce a `LoweringError` per spec §14.3
  4. `on create`, `on interact(who)`, and `on destroy` lifecycle hooks lower to lifecycle hook registrations; `[Singleton]` attribute is recognized and propagated
  5. Snapshot tests cover: component field flattening with partial overrides, lifecycle hooks, `[Singleton]` attribute, conflicting component method names

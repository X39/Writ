**4. Migration error for old construction syntax:**

After removing BraceConstruct, when the parser encounters `Ident { Ident:` (which used to be construction), it will produce a generic error. To improve diagnostics, we don't need to add custom error recovery in the parser combinator itself — the removal of BraceConstruct will naturally cause `Type { field: value }` to fail to parse, and the error will point at the `{`. The targeted "construction requires `new` keyword" error is a nice-to-have that can be implemented as a custom error label if desired, but the primary requirement is that the old syntax is rejected. The CONTEXT.md says this is a parser-level diagnostic — so add a `.labelled("expression")` on the atom choice or use a custom recovery to produce a better message.

For a pragmatic approach: after the atom choice, add a recovery rule that detects the `Ident { Ident :` pattern and produces a custom error. This can be done with a `.recover_with()` strategy or by adding a "trap" parser that matches the pattern and emits a custom error via `Parser::validate()`.

The simplest effective approach: add a comment noting the migration error behavior, and rely on the parser's existing error reporting — when `Type { field: }` is encountered, the `{` will be unexpected since BraceConstruct is removed, and chumsky will report the error location. A user can infer they need `new`. Full custom error messaging is a polish item.

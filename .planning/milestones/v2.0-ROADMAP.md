### 🚧 v2.0 Writ Runtime (In Progress)

**Milestone Goal:** Build a spec-compliant IL runtime (VM, module loader, assembler) as a reference implementation to validate future AST-to-IL codegen.

- [x] **Phase 16: Module Format Foundation** - `writ-module` crate with binary reader/writer, programmatic IL builder API, and full 98-opcode instruction enum
- [x] **Phase 17: VM Core and Task Execution** - Register-based dispatch loop, managed call frame stack, cooperative task scheduler, suspend-and-confirm, defer/crash, atomic sections, and RuntimeHost trait
- [x] **Phase 18: Entity System and GC** - Generation-indexed entity registry with lifecycle hooks, precise mark-and-sweep GC with trait boundary, and component suspend-and-confirm integration
- [x] **Phase 19: Contract Dispatch and Virtual Module** - CALL_VIRT contract dispatch table, `writ-runtime` virtual module (Option, Result, Range, 17 contracts, primitives, Array, Entity), cross-module name resolution
- [x] **Phase 20: Text Assembler** - `writ-assembler` crate: text IL format, two-pass assembler producing spec-valid binary modules
- [x] **Phase 21: Disassembler and Runner CLI** - Binary-to-text disassembler, `writ-runner` CLI with NullHost and end-to-end integration test (completed 2026-03-02)

## Phase Details

### Phase 16: Module Format Foundation
**Goal**: The `writ-module` crate exists as a standalone pure-data crate; any IL module can be written to bytes, read back from bytes, and produced programmatically without touching VM logic
**Depends on**: Nothing (first v2.0 phase)
**Requirements**: MOD-01, MOD-02, MOD-03, MOD-04, MOD-05, MOD-06
**Success Criteria** (what must be TRUE):
  1. A spec-compliant binary module (200-byte header, all 21 metadata tables, string/blob heaps, method bodies) can be read from disk and its tables inspected in tests
  2. A module constructed with `ModuleBuilder` can be serialized to bytes that a spec-compliant reader accepts without error
  3. Writing a module, reading it back, and writing it again produces identical bytes (round-trip identity)
  4. `MetadataToken(0)` is the null token; all table lookups through the newtype return `None` for 0 and `Some(row)` for n >= 1 — confirmed by test
  5. All 91 opcodes encode and decode without information loss, verified by round-trip test across the full opcode table
**Plans**: 3 plans

Plans:
- [x] 16-01-PLAN.md -- Crate setup + data model (MetadataToken, 21 table row structs, 98-opcode Instruction enum, Module struct, heap utilities)
- [x] 16-02-PLAN.md -- Binary reader + writer + round-trip identity tests
- [x] 16-03-PLAN.md -- ModuleBuilder fluent API + builder integration tests

### Phase 17: VM Core and Task Execution
**Goal**: The VM executes IL instructions against a typed register file using a managed call frame stack; the cooperative task scheduler drives tasks through all 5 lifecycle states with correct suspend-and-confirm, defer unwinding, crash propagation, and atomic section enforcement
**Depends on**: Phase 16
**Requirements**: VM-01, VM-02, VM-03, VM-04, VM-05, VM-06, VM-07, TASK-01, TASK-02, TASK-03, TASK-04, TASK-05, TASK-06, TASK-07, TASK-08
**Success Criteria** (what must be TRUE):
  1. All 91 instructions execute in a single match-dispatch loop; arithmetic, comparison, logic, control flow, string conversion, and object model instructions produce spec-correct results verified by unit tests
  2. A task advances through Ready -> Running -> Suspended -> Completed states correctly; SPAWN/JOIN/CANCEL manage child tasks without corrupting the scheduler
  3. A deferred call stack executes in LIFO order on function return, task cancel, and crash; secondary crashes inside defer handlers are swallowed and logged
  4. ATOMIC_BEGIN/ATOMIC_END prevents any other task from running between them, verified by a two-task interleaving test
  5. The `RuntimeHost` trait compiles with `NullHost` as a synchronous no-op implementation; all 9 suspend-and-confirm transition points correctly suspend the task and resume after host confirmation
**Plans**: 3 plans

Plans:
- [x] 17-01-PLAN.md -- Foundation types, heap, module loader, RuntimeHost/NullHost
- [x] 17-02-PLAN.md -- Dispatch loop (all 91 instructions), call frame management, scheduler, Runtime API, per-instruction tests
- [x] 17-03-PLAN.md -- Defer/crash engine, atomic sections, concurrency instructions, task lifecycle tests

### Phase 18: Entity System and GC
**Goal**: Entities are created, destroyed, and queried through generation-indexed handles with no stale-handle aliasing; lifecycle hooks fire at correct points; a precise mark-and-sweep GC collects unreachable objects at safe points without running finalizer IL inside the collection sweep
**Depends on**: Phase 17
**Requirements**: ENT-01, ENT-02, ENT-03, ENT-04, ENT-05, ENT-06, ENT-07, ENT-08, GC-01, GC-02, GC-03, GC-04, GC-05
**Success Criteria** (what must be TRUE):
  1. Accessing an entity through a stale handle (after destroy or collection) returns an error without undefined behavior, confirmed by test
  2. `SPAWN_ENTITY` followed by field sets and `INIT_ENTITY` triggers `on_create`; the entity is visible to the host only after `INIT_ENTITY` completes
  3. `on_destroy` fires before host notification; `on_interact(who)` dispatches as a new independent task; singleton GET_OR_CREATE returns the existing entity on second call
  4. Unreachable heap objects are reclaimed after the GC runs; `on_finalize` hooks fire as scheduled tasks after the sweep completes, not during collection
  5. Swapping the GC implementation requires changing only the `trait GcHeap` implementor — no VM dispatch loop changes
**Plans**: 3 plans

Plans:
- [x] 18-01-PLAN.md -- GcHeap trait, BumpHeap adapter, EntityRegistry with generation-indexed handles
- [x] 18-02-PLAN.md -- Dispatch loop migration to GcHeap, entity instruction handlers, lifecycle hooks
- [x] 18-03-PLAN.md -- MarkSweepHeap GC, root collection, finalization queue, Runtime::collect_garbage()

### Phase 19: Contract Dispatch and Virtual Module
**Goal**: `CALL_VIRT` resolves to the correct method implementation at O(1) via a dispatch table built at load time; the `writ-runtime` virtual module is available at domain startup providing all standard types, contracts, and primitives without reading a file from disk
**Depends on**: Phase 18
**Requirements**: DISP-01, DISP-02, DISP-03, DISP-04, DISP-05, DISP-06
**Success Criteria** (what must be TRUE):
  1. `CALL_VIRT` on a concrete type dispatches to the correct `ImplDef` method without a linear scan; incorrect type/contract combinations produce a runtime error, not a panic
  2. `Option<T>`, `Result<T,E>`, and `Range<T>` are available as types in any loaded module without requiring an external file
  3. All 17 contracts defined in the spec are present in the virtual module and resolvable by name from user modules
  4. Primitive types (Int, Float, Bool, String) have pseudo-TypeDefs in the virtual module; their intrinsic methods dispatch correctly through the contract table
  5. A `TypeRef`, `MethodRef`, or `FieldRef` in one module resolves correctly to the matching definition in another loaded module; unresolvable references produce a named error at load time
**Plans**: 3 plans

Plans:
- [x] 19-01-PLAN.md -- Virtual module construction (17 contracts, Option/Result/Range, primitive pseudo-TypeDefs, Array, Entity)
- [x] 19-02-PLAN.md -- Cross-module resolution infrastructure (Domain, TypeRef/MethodRef/FieldRef resolution)
- [x] 19-03-PLAN.md -- Dispatch table and CALL_VIRT wiring (HashMap dispatch, intrinsic routing, Runtime migration)

### Phase 20: Text Assembler
**Goal**: A developer can write human-readable text IL, run the assembler, and get a spec-valid binary module that the VM can execute; the assembler correctly resolves forward label references in a two-pass process
**Depends on**: Phase 16 (for ModuleBuilder; can be built in parallel with Phases 17-19 but integration-tested after Phase 17)
**Requirements**: ASM-01, ASM-02, ASM-03, ASM-04
**Success Criteria** (what must be TRUE):
  1. A text IL file with `.module`, `.type`, `.field`, `.contract`, `.impl`, `.method`, and `.reg` directives assembles to a binary module without error
  2. A forward label reference (jump to a label defined later in the method body) resolves correctly in the output binary
  3. The assembled binary passes the module reader without error and is accepted by the VM for execution
  4. A text IL source can be assembled to binary, disassembled back to text, and the disassembled text re-assembles to identical bytes
**Plans**: 2 plans

Plans:
- [x] 20-01-PLAN.md -- Crate setup + lexer + AST + recursive-descent parser with multi-error collection
- [x] 20-02-PLAN.md -- Two-pass assembler (name resolution, label resolution, ModuleBuilder integration) + integration tests

### Phase 21: Disassembler and Runner CLI
**Goal**: A developer can disassemble any binary IL module to readable text; a developer can run a complete IL module from the command line and see `say()` output on stdout; end-to-end stack validation is confirmed by executing a real dialogue module
**Depends on**: Phases 16-20
**Requirements**: TOOL-01, TOOL-02, TOOL-03, TOOL-04
**Success Criteria** (what must be TRUE):
  1. `writ disasm module.writc` prints a human-readable text representation of every type, field, method, and instruction in the module
  2. `writ run module.writc` executes the module's entry task to completion; `say()` calls appear on stdout; the process exits 0 on success
  3. `CliHost` handles all host interface calls with annotated output; `NullHost` stays untouched as testing-only host
  4. All three subcommands (`run`, `assemble`, `disasm`) are available from the `writ` binary and produce useful error messages for invalid input
**Plans**: 2 plans

Plans:
- [x] 21-01-PLAN.md -- Disassembler implementation in writ-assembler (binary Module to .writil text, round-trip tests)
- [ ] 21-02-PLAN.md -- CLI binary with clap, CliHost, runner loop, integration tests

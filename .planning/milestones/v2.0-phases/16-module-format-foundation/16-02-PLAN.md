<objective>
Implement the binary reader (`Module::from_bytes`) and writer (`Module::to_bytes`) with round-trip identity verification. The reader parses the 200-byte header, reads all 21 metadata tables, reads string/blob heaps, and decodes method body instruction streams. The writer serializes all of this back to bytes. Round-trip identity (write -> read -> write = identical bytes) is the primary correctness criterion.

Purpose: Enable loading and saving spec-compliant IL binary modules, which is the core I/O capability that the VM (Phase 17), assembler (Phase 20), and disassembler (Phase 21) depend on.
Output: `Module::from_bytes(bytes)` and `module.to_bytes()` methods with comprehensive round-trip tests.
</objective>

  <files>

  <behavior>
    - from_bytes on bytes starting with "WRIT" + valid header parses successfully
    - from_bytes on bytes starting with "XRIT" returns DecodeError::BadMagic
    - from_bytes on truncated input returns an IO/EOF-related error
    - to_bytes on an empty Module produces a valid 200-byte header with zero row counts
    - to_bytes on a module with populated tables produces bytes that from_bytes can read back
    - Module with string heap entries round-trips: heap offsets and strings are preserved
    - Module with method bodies round-trips: register types, code bytes, and decoded instructions match
  </behavior>
  <action>
    1. Create `writ-module/src/reader.rs`:
       - `pub fn from_bytes(bytes: &[u8]) -> Result<Module, DecodeError>`
       - Wrap bytes in `Cursor<&[u8]>`
       - Read magic bytes (4 bytes), check == b"WRIT" or return BadMagic
       - Read format_version (u16 LE), flags (u16 LE)
       - Read module_name offset (u32), module_version offset (u32)
       - Read string_heap_offset (u32), string_heap_size (u32), blob_heap_offset (u32), blob_heap_size (u32)
       - Read table directory: 21 entries of (offset: u32, row_count: u32)
       - Total header: 200 bytes. Assert cursor position == 200 after header read.
       - Read string heap: slice bytes[string_heap_offset..+string_heap_size], clone into Vec<u8>
       - Read blob heap: slice bytes[blob_heap_offset..+blob_heap_size], clone into Vec<u8>
       - For each of 21 tables with row_count > 0: seek to table offset, read row_count rows using table-specific deserialization. Each row struct has fixed-size fields — read them sequentially with ReadBytesExt. Table row order matches TableId enum order (0=ModuleDef through 20=AttributeDef).
       - **Table row sizes (compute from field widths per §2.16.5):**
         - ModuleDef: 12 bytes (u32 + u32 + u32)
         - ModuleRef: 8 bytes (u32 + u32)
         - TypeDef: 19 bytes (u32 + u32 + u8 + u16 + u32 + u32) — BUT align rows to 4 bytes → pad to 20 bytes
         - TypeRef: 12 bytes (u32 + u32 + u32)
         - TypeSpec: 4 bytes (u32)
         - FieldDef: 10 bytes → pad to 12
         - FieldRef: 12 bytes (u32 + u32 + u32)
         - MethodDef: 20 bytes (u32 + u32 + u16 + u32 + u32 + u16) → pad to 20
         - MethodRef: 12 bytes
         - ParamDef: 10 bytes → pad to 12
         - ContractDef: 16 bytes
         - ContractMethod: 10 bytes → pad to 12
         - ImplDef: 12 bytes
         - GenericParam: 11 bytes → pad to 12
         - GenericConstraint: 8 bytes
         - GlobalDef: 14 bytes → pad to 16
         - ExternDef: 14 bytes → pad to 16
         - ComponentSlot: 8 bytes
         - LocaleDef: 12 bytes
         - ExportDef: 9 bytes → pad to 12
         - AttributeDef: 13 bytes → pad to 16
       - **IMPORTANT:** The spec says "Table rows are aligned to 4-byte boundaries" (§2.16.1). For rows whose natural field sum is not a multiple of 4, read padding bytes after the fields. Define a constant ROW_SIZE for each table that is the 4-byte-aligned size.
       - Read method bodies: for each MethodDefRow where body_size > 0, seek to body_offset, read register_types (reg_count u32 values), read code_size (u32), read code bytes, decode instructions from code stream. If debug flag is set in module flags, read debug info.
       - Store decoded instructions as a `Vec<Instruction>` in the MethodBody alongside the raw `code: Vec<u8>`. The raw code bytes enable byte-exact round-trip; decoded instructions are for inspection.

    2. Create `writ-module/src/writer.rs`:
       - `pub fn to_bytes(module: &Module) -> Result<Vec<u8>, EncodeError>`
       - Phase 1: Compute sizes. Calculate each table's byte size (row_count * ROW_SIZE_ALIGNED). Calculate heap sizes. Calculate total file size.
       - Phase 2: Write header (200 bytes). Magic, version, flags, name/version offsets, heap offsets/sizes, table directory with computed offsets.
       - Phase 3: Write tables in spec order (0-20). Each row written with LE fields using WriteBytesExt. Pad rows to 4-byte alignment.
       - Phase 4: Write heaps (string heap bytes verbatim, then blob heap bytes verbatim).
       - Phase 5: Write method bodies. For each MethodDefRow with body_size > 0, write register type u32s, write code_size, write raw code bytes. Pad method body start offsets to 4-byte alignment between bodies.
       - **CRITICAL for round-trip:** Write heap bytes verbatim from Module.string_heap/blob_heap — do NOT re-intern or reorder. Write raw code bytes from MethodBody.code — do NOT re-encode instructions. This guarantees byte-exact round-trip.
       - Compute table and heap offsets: tables start at byte 200 (after header). Heaps follow tables. Method bodies follow heaps.

    3. Update `writ-module/src/module.rs`:
       - Add `pub fn from_bytes(bytes: &[u8]) -> Result<Module, DecodeError>` that delegates to `reader::from_bytes`
       - Add `pub fn to_bytes(&self) -> Result<Vec<u8>, EncodeError>` that delegates to `writer::to_bytes`

    4. Update `writ-module/src/lib.rs`:
       - Add `mod reader;` and `mod writer;` (private modules — public API is on Module)

    **Key implementation detail for row serialization:**
    Define a trait or helper functions for reading/writing each row type. Each row type needs a `read_row(cursor) -> Result<Row>` and `write_row(row, writer) -> Result<()>`. Keep these as private functions in reader.rs/writer.rs respectively. Use a match on TableId to dispatch to the correct reader/writer.

    **Layout computation order:**
    ```
    Offset 0:     Header (200 bytes)
    Offset 200:   Tables (in order 0-20, each row_count * aligned_row_size)
    After tables: String heap
    After string: Blob heap
    After blob:   Method bodies (each 4-byte aligned)
    ```
    The header's table_directory[i] stores the absolute offset of table i. The header's string_heap_offset/blob_heap_offset store absolute offsets. MethodDef.body_offset stores absolute offset.

<objective>
Implement the ModuleBuilder API for programmatic construction of IL modules. The builder provides a fluent API where users work with string values (not heap offsets), and `build()` produces a complete Module with properly interned heaps. This is the primary API for test authoring and will be used by the assembler (Phase 20) and compiler backend.

Purpose: Enable ergonomic programmatic construction of IL modules for testing and tooling without manually managing heap offsets or table indices.
Output: `ModuleBuilder` with fluent methods and comprehensive tests proving builder-produced modules are valid.
</objective>

        pub fn new(name: &str) -> Self { /* initialize with name, version "1.0.0", empty vecs */ }
        pub fn version(mut self, v: &str) -> Self { self.version = v.to_string(); self }

        // Each add_* method pushes to the corresponding Vec and returns the MetadataToken
        // for the new row. Token table_id matches the TableId enum.
        // Row index is 1-based (vec.len() after push).

        pub fn add_type_def(&mut self, name: &str, namespace: &str, kind: u8, flags: u16) -> MetadataToken {
            let idx = self.type_defs.len() as u32 + 1; // 1-based
            self.type_defs.push(TypeDefBuilder { name: name.to_string(), namespace: namespace.to_string(), kind, flags, field_list: 0, method_list: 0 });
            MetadataToken::new(TableId::TypeDef.as_u8(), idx)
        }

        pub fn add_impl_def(&mut self, type_token: MetadataToken, contract: MetadataToken) -> MetadataToken { ... }

    3. Implement `build(self) -> Module`:
       - Initialize string heap and blob heap via `heap::init_string_heap()` and `heap::init_blob_heap()`
       - Intern module name and version strings into the string heap
       - For each builder row, intern its String fields into the string heap (getting u32 offsets) and Vec<u8> fields into the blob heap
       - Convert each XxxBuilder to the corresponding XxxRow with the interned offsets
       - Create ModuleDefRow with the interned name/version offsets and flags=0
       - For method bodies with instructions already encoded to code bytes, compute body_offset and body_size (these will be finalized by the writer, but set reasonable placeholder values — the writer recomputes them during to_bytes)
       - Construct and return `Module { header, string_heap, blob_heap, module_defs: vec![module_def_row], ...all tables..., method_bodies }`
       - Set header.format_version = 1, header.flags = 0
       - Header offset fields (string_heap_offset, etc.) will be computed by the writer during to_bytes, so set them to 0 in the builder — the writer is authoritative for layout

    Note on field_list/method_list in TypeDef: The list ownership pattern means TypeDef.field_list = index of first FieldDef belonging to this type. The builder should track which fields/methods belong to which type. A simple approach: the user adds types first, then adds fields/methods. The builder tracks the current field/method count and sets field_list/method_list appropriately. For Phase 16, a simpler approach is acceptable: expose set_field_list/set_method_list methods, or compute them in build() based on insertion order. Document the expected usage pattern in doc comments.

    Simpler approach for Phase 16: The `add_type_def` method records the current length of field_defs and method_defs as the field_list and method_list values. Users must add a type's fields and methods immediately after adding the type, before adding another type. This matches the spec's "list ownership" pattern where ranges are implicit.

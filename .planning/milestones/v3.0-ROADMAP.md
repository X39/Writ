## Phase Details

### Phase 22: Name Resolution
**Goal**: Every identifier in every source file is bound to a declaration; the compiler produces a NameResolved IR where no identifier is an unresolved string
**Depends on**: Phase 21 (v2.0 complete; resolve consumes AST from writ-compiler lowering pipeline)
**Requirements**: RES-01, RES-02, RES-03, RES-04, RES-05, RES-06, RES-07, RES-08, RES-09, RES-10, RES-11, RES-12
**Success Criteria** (what must be TRUE):
  1. Compiler collects all top-level declarations across all source files into a symbol table before resolving any reference in any body (two-pass, no forward-reference failures)
  2. Every `using` import resolves or produces an error with the conflicting-candidate spans; every `::` qualified path resolves to its target declaration
  3. Every `AstType` in every declaration and body resolves to a TypeRef blob or primitive tag, including types from the writ-runtime virtual module
  4. Visibility violations produce errors that name both the access site and the declaration's visibility modifier
  5. `@Speaker` names that do not resolve to a `[Singleton]` entity with a Speaker component produce a named resolution error; `[Singleton]`/`[Conditional]` on disallowed declaration kinds produce attribute errors
**Plans**: 3 plans

Plans:
- [x] 22-01-PLAN.md — writ-diagnostics crate, DefMap, Pass 1 declaration collector, prelude, config, IR types (Wave 1)
- [x] 22-02-PLAN.md — ScopeChain, Pass 2 body resolver, using/path/visibility/type resolution, impl association, generics, self (Wave 2)
- [x] 22-03-PLAN.md — Speaker validation, attribute validation, ambiguity reporting, fuzzy "did you mean?" suggestions (Wave 3)

### Phase 23: Type Checking
**Goal**: Every expression node in the program carries a fully resolved, non-optional type; all type rules from the Writ spec are enforced with precise, actionable errors
**Depends on**: Phase 22 (requires complete DefMap with every identifier bound to a DefId)
**Requirements**: TYPE-01, TYPE-02, TYPE-03, TYPE-04, TYPE-05, TYPE-06, TYPE-07, TYPE-08, TYPE-09, TYPE-10, TYPE-11, TYPE-12, TYPE-13, TYPE-14, TYPE-15, TYPE-16, TYPE-17, TYPE-18, TYPE-19
**Success Criteria** (what must be TRUE):
  1. The type checker produces a Typed IR where every expression node carries a non-optional Ty; no `Option<Ty>` fields exist in the typed representation
  2. `let` bindings infer their type from the initializer; annotated bindings are checked for compatibility; contract bound violations at generic call sites are reported with the unsatisfied bound named
  3. `let` immutability is enforced at both reassignment sites AND mutation-through-immutable-binding sites (two distinct checks, both producing errors pointing to the binding and the violation)
  4. `?` and `!` operators are desugared to typed match nodes in the Typed IR; the Typed IR contains no raw `UnaryPostfix` nodes
  5. Enum `match` exhaustiveness is checked; missing variants are named in the error; all code paths returning a non-void type are verified to reach a typed return
**Plans**: 3 plans

Plans:
- [x] 23-01-PLAN.md — Typed IR foundation, TyInterner, TypeEnv, let inference, function call checking (Wave 1)
- [x] 23-02-PLAN.md — Field access, component access, contract bounds, mutability enforcement, return type checking (Wave 2)
- [x] 23-03-PLAN.md — ?/!/try desugaring, enum exhaustiveness, closure captures, spawn/join/cancel, new/for typing (Wave 3)

### Phase 24: IL Codegen — Metadata Skeleton
**Goal**: All 21 metadata tables are populated in the ModuleBuilder with correct token assignment; every DefId has a MetadataToken before any method body is emitted
**Depends on**: Phase 23 (requires Typed IR with all types resolved; CALL_VIRT slot ordering from contract declaration is established here)
**Requirements**: EMIT-01, EMIT-02, EMIT-03, EMIT-04, EMIT-05, EMIT-06, EMIT-22, EMIT-25, EMIT-29
**Success Criteria** (what must be TRUE):
  1. The compiler produces a ModuleBuilder with all TypeDef, FieldDef, MethodDef, ParamDef, ContractDef, ContractMethod, and ImplDef rows populated; every row has a stable MetadataToken assigned
  2. CALL_VIRT slot numbers are derived from contract declaration order, not impl block traversal order; the slot assignment is fixed before any method body emits a CALL_VIRT instruction
  3. Entity TypeDefs have lifecycle hook MethodDef tokens registered in their hook_kind slots (on_create, on_destroy, on_interact, on_finalize each registered where present)
  4. LocaleDef rows are emitted for all localization keys generated during lowering
**Plans**: 2 plans

Plans:
- [x] 24-01-PLAN.md — Emit module foundation, ModuleBuilder, MetadataToken, heaps, TypeDef/FieldDef/MethodDef/ParamDef/GenericParam/GenericConstraint/ModuleDef/ModuleRef/ExportDef emission (Wave 1)
- [x] 24-02-PLAN.md — ContractDef/ContractMethod/ImplDef/GlobalDef/ExternDef/ComponentSlot/LocaleDef/AttributeDef emission, CALL_VIRT slot assignment (Wave 2)

### Phase 25: IL Codegen — Method Bodies
**Goal**: All method bodies emit correct, spec-compliant instruction sequences; every IL instruction is selected by the compiler's type annotations; the output .writil module loads and executes on the VM
**Depends on**: Phase 24 (requires complete DefId-to-MetadataToken map; all metadata rows must exist before body instructions reference them)
**Requirements**: EMIT-07, EMIT-08, EMIT-09, EMIT-10, EMIT-11, EMIT-12, EMIT-13, EMIT-14, EMIT-15, EMIT-16, EMIT-17, EMIT-18, EMIT-19, EMIT-20, EMIT-21, EMIT-23, EMIT-24, EMIT-26, EMIT-27, EMIT-28
**Success Criteria** (what must be TRUE):
  1. Register allocation uses a LIFO high-watermark allocator; no two simultaneously live values share a register; the per-register TypeRef table is emitted correctly
  2. Entity construction follows the spec §14.7.5 sequence exactly: `SPAWN_ENTITY` then `SET_FIELD` only for explicitly-provided field overrides, then `INIT_ENTITY`; default field values do not generate SET_FIELD instructions
  3. Closure/delegate emission produces a compiler-generated capture struct TypeDef, a method body, and a NEW_DELEGATE instruction; closure captures are correctly classified as by-value or by-reference per the type checker's annotations
  4. `?` and `try` propagation emit IS_NONE/IS_ERR plus early-return instruction sequences; dialogue transition returns emit TAIL_CALL; BOX/UNBOX are emitted at generic call sites where the type checker annotated boxing
  5. Debug info (SourceSpan and DebugLocal entries) is emitted for all method bodies; constant arithmetic in `const` declarations is folded at compile time
**Plans**: 6 plans

Plans:
- [x] 25-01-PLAN.md — Register allocator, labels, BodyEmitter infrastructure, core instruction emission (arithmetic/logic/comparison/data-movement/control-flow) (Wave 1)
- [x] 25-02-PLAN.md — Call dispatch (CALL/CALL_VIRT/CALL_EXTERN/CALL_INDIRECT), CALL_VIRT specialization, object model (NEW/GET_FIELD/SET_FIELD), entity construction, BOX/UNBOX (Wave 2)
- [x] 25-03-PLAN.md — Arrays, Option/Result, closures/delegates with capture struct synthesis, concurrency (SPAWN_TASK/JOIN/CANCEL/DEFER), atomic blocks (Wave 3)
- [x] 25-04-PLAN.md — Enum match/construction, type conversions, string ops, ?/try propagation, TAIL_CALL, debug info, constant folding, binary serialization to Vec<u8> (Wave 4)
- [x] 25-05-PLAN.md — Gap closure: SWITCH offset fixup, closure body emission, const_fold wiring, string literal interning (Wave 5)
- [x] 25-06-PLAN.md — Gap closure: TAIL_CALL for dialogue transitions, StrBuild for multi-part string chains (Wave 6)

### Phase 26: CLI Integration and End-to-End Validation
**Goal**: `writ compile` accepts Writ source files and produces a .writil binary that the existing VM can load and execute; the full pipeline is validated against representative Writ programs
**Depends on**: Phase 25 (requires complete codegen output producing valid .writil modules); Phase 22-24 (runtime gaps must be fixed before e2e programs exercise lifecycle hooks and generic dispatch)
**Requirements**: CLI-01, CLI-02, CLI-03, FIX-01, FIX-02, FIX-03
**Success Criteria** (what must be TRUE):
  1. `writ compile <file.writ>` produces a .writil binary module; `writ run <file.writil>` executes it; the source-to-execution pipeline works end-to-end without manual assembly steps
  2. Compilation errors display with source spans, multi-span context (ariadne rendering), and actionable messages; the error format matches the quality of existing parse errors
  3. Runtime dispatches on_create, on_destroy, on_interact, and on_finalize lifecycle hooks by method name lookup; the hooks fire at the correct entity lifecycle events
  4. Generic contract specialization (e.g., `Int:Into<Float>` vs `Int:Into<String>`) resolves without dispatch table collision; the correct impl is selected at runtime
**Plans**: 4 plans

Plans:
- [x] 26-01-PLAN.md — Runtime gap closure: lifecycle hook dispatch, generic dispatch key, CliHost string deref (Wave 1)
- [x] 26-02-PLAN.md — `writ compile` subcommand, pipeline wiring, LoweringError diagnostics, ariadne rendering (Wave 1)
- [x] 26-03-PLAN.md — End-to-end integration tests: compile + run validation (Wave 2)
- [x] 26-04-PLAN.md — Gap closure: FIX-02 generic dispatch collision resolution (Wave 1)

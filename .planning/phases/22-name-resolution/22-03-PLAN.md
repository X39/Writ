    Create `writ-compiler/src/resolve/suggest.rs`:

    **suggest_similar function:**
    ```rust
    pub fn suggest_similar(
        unresolved: &str,
        visible_names: impl Iterator<Item = (String, bool)>,  // (name, is_imported_via_using)
        all_names: impl Iterator<Item = (String, String)>,    // (simple_name, fqn)
    ) -> Vec<Suggestion>
    ```

    **Suggestion struct:**
    ```rust
    pub struct Suggestion {
        pub name: String,           // the suggested name
        pub fqn: Option<String>,    // fully-qualified name if different from name
        pub needs_import: bool,     // true if requires `using` to access
        pub import_ns: Option<String>, // namespace to import if needs_import
        pub score: f64,             // similarity score for sorting
    }
    ```

    **Algorithm:**
    1. Use `strsim::jaro_winkler(unresolved, candidate)` for scoring.
    2. Threshold: `>= 0.8` (tunable constant `SIMILARITY_THRESHOLD`).
    3. Max suggestions: 3 (constant `MAX_SUGGESTIONS`).
    4. Two-phase search:
       a. **Phase A: visible names** — names already in scope (locals, current namespace, usings, prelude). These produce simple "did you mean `X`?" suggestions.
       b. **Phase B: all names** — names from ALL namespaces in the DefMap (cross-namespace search). These produce "did you mean `ns::X`? (add `using ns;`)" suggestions. Only include Phase B results if Phase A produced fewer than MAX_SUGGESTIONS.
    5. Sort by score descending, take top MAX_SUGGESTIONS.
    6. Include primitives and prelude names in the candidate pool (Phase A always).

    **Integrate into ScopeChain (update scope.rs):**
    When `resolve_type()` or `resolve_value()` fails to find a name:
    1. Collect all visible names from the current scope (generics, locals, current namespace members, using imports, prelude).
    2. Collect all names from the entire DefMap (for cross-namespace suggestions).
    3. Call `suggest_similar(unresolved, visible, all)`.
    4. Attach suggestions to the `UnresolvedName` / `UnresolvedType` diagnostic's `help` field.

    **Format suggestion into help text:**
    - If `needs_import == false`: `help: "did you mean `{name}`?"`
    - If `needs_import == true`: `help: "did you mean `{fqn}`? (add `using {import_ns};`)"`
    - Multiple suggestions: join with newlines or "or" separator.
    - Example: `help: "did you mean `survival::HealthPotion`? (add `using survival;`)\n  or: `combat::HealthPotion`? (add `using combat;`)"`

    **Desugared syntax tracking (CONTEXT.md requirement):**
    When reporting an error about a type that was desugared (e.g., `T?` → `Option<T>`, `T[]` → `Array<T>`), include the original syntax in parentheses. This requires the resolver to track the original AstType that produced the resolution attempt.

    Implementation: In `resolve_type()`, when resolving `AstType::Generic { name: "Option", args, .. }` check if the span corresponds to a `?` suffix (the span will be tight around the `?`). If the generic name is "Option" and it has exactly 1 arg and the original source at the span position contains `?`, add a note: "(written as `{inner_type}?`)" to the error message.

    Similarly for `AstType::Array { .. }` → note "(written as `{elem_type}[]`)".

    This is best-effort; if the original syntax cannot be detected, omit the note.

    **Update error.rs:**
    Enhance `UnresolvedType` and `UnresolvedName` error variants to include `suggestions: Vec<Suggestion>` field. Update the `From<ResolutionError> for Diagnostic` implementation to format suggestions into the help text.

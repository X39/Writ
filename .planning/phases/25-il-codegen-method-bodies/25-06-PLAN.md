**TAIL_CALL emission (EMIT-24):**

Dialogue transitions are lowered to regular `Return(Call(...))` patterns at the AST level. The TypedAST has no explicit dialogue flag. The correct approach: detect the tail-call pattern `Return { value: Some(Call { ... }) }` and emit `TailCall` instead of `Call + Ret`.

1. In `emit_expr()` for `TypedExpr::Return`, detect when the value is a `TypedExpr::Call`:

```rust
TypedExpr::Return { value, ty, .. } => {
    if let Some(inner) = value {
        // Check for tail-call pattern: Return(Call(...))
        if let TypedExpr::Call { callee, args, ty: call_ty, .. } = inner.as_ref() {
            return emit_tail_call(emitter, callee, args);
        }
        // Normal return
        let r_val = emit_expr(emitter, inner);
        emitter.emit(Instruction::Ret { r_src: r_val });
        r_val
    } else {
        let r_void = emitter.alloc_void_reg();
        emitter.emit(Instruction::Ret { r_src: r_void });
        r_void
    }
}
```

2. Create `emit_tail_call()` helper function:

```rust
fn emit_tail_call(emitter: &mut BodyEmitter<'_>, callee: &TypedExpr, args: &[TypedExpr]) -> u16 {
    // Emit arguments into consecutive registers
    let r_base = emitter.regs.next();
    let arg_regs: Vec<u16> = args.iter().map(|arg| emit_expr(emitter, arg)).collect();

    // Pack arguments into consecutive block starting at r_base
    for (i, &r_arg) in arg_regs.iter().enumerate() {
        let r_slot = r_base + i as u16;
        if r_arg != r_slot {
            emitter.emit(Instruction::Mov { r_dst: r_slot, r_src: r_arg });
        }
    }

    // Resolve method_idx from callee (same pattern as Call dispatch)
    // For now, use 0 as placeholder method_idx (same as existing Call patterns)
    // The full pipeline will resolve this via DefId -> MetadataToken
    let method_idx = 0u32;

    emitter.emit(Instruction::TailCall {
        method_idx,
        r_base,
        argc: args.len() as u16,
    });

    // TailCall doesn't return to this frame, but we need a register to satisfy the return type
    emitter.alloc_void_reg()
}
```

3. Also check `TypedStmt::Return` in stmt.rs for the same pattern. If stmt.rs has a Return handler that delegates to expr.rs, this may already be covered. If not, add the same tail-call detection there.

**StrBuild emission (EMIT-20 completeness):**

Format strings are lowered to left-associative `BinaryOp::Add` chains where every node has `TyKind::String`. A chain like `a + b + c + d` becomes:
```
Binary(Add, Binary(Add, Binary(Add, a, b), c), d)
```

The optimization: when the result type is String and we detect a chain of 3+ Add operations all on strings, emit a single `StrBuild { r_dst, count, r_base }` with all parts in consecutive registers, instead of nested `StrConcat` calls.

1. Create a helper `collect_string_add_chain()` that recursively flattens a left-associative Add chain into a flat list of leaf operands:

```rust
fn collect_string_add_chain<'a>(expr: &'a TypedExpr, parts: &mut Vec<&'a TypedExpr>) {
    match expr {
        TypedExpr::Binary { left, op: BinaryOp::Add, right, ty, .. }
            if matches!(emitter.interner.kind(*ty), TyKind::String) =>
        {
            collect_string_add_chain(left, parts);
            parts.push(right);
        }
        _ => {
            parts.push(expr);
        }
    }
}
```

Note: This function needs access to the interner for type checking. Either pass the interner as a parameter or make it a method on BodyEmitter. Simplest: make it a standalone function taking `&TyInterner`.

2. In `emit_binary()`, when `op == BinaryOp::Add` and `ty_kind == TyKind::String`:

```rust
TyKind::String => {
    // Check if this is a multi-part string chain (3+ parts)
    let mut parts = Vec::new();
    // The current Binary node is (left + right) — collect leaf parts
    collect_string_add_chain_with_interner(left_expr, right_expr, emitter.interner, &mut parts);

    if parts.len() >= 3 {
        // Emit StrBuild with all parts in consecutive registers
        let r_base = emitter.regs.next();
        let part_regs: Vec<u16> = parts.iter().map(|p| emit_expr(emitter, p)).collect();
        // Pack into consecutive block
        for (i, &r_part) in part_regs.iter().enumerate() {
            let r_slot = r_base + i as u16;
            if r_part != r_slot {
                emitter.emit(Instruction::Mov { r_dst: r_slot, r_src: r_part });
            }
        }
        let r_dst = emitter.alloc_reg(ty);
        emitter.emit(Instruction::StrBuild { r_dst, count: parts.len() as u16, r_base });
        r_dst
    } else {
        // 2 parts: use StrConcat as before
        let r_dst = emitter.alloc_reg(ty);
        emitter.emit(Instruction::StrConcat { r_dst, r_a, r_b });
        r_dst
    }
}
```

**IMPORTANT:** The current `emit_binary()` receives pre-emitted `r_a` and `r_b` registers, NOT the original expressions. To collect the chain, we need access to the original TypedExpr nodes. This means the StrBuild optimization must be detected BEFORE emitting sub-expressions.

Approach: In `emit_expr()` for `TypedExpr::Binary`, before calling `emit_binary()`, check if the binary is a string Add chain of 3+ parts. If so, handle it directly in `emit_expr()` instead of going through `emit_binary()`:

```rust
TypedExpr::Binary { left, op, right, ty, .. } => {
    // Check for multi-part string concatenation chain BEFORE emitting sub-exprs
    if *op == BinaryOp::Add {
        if let TyKind::String = emitter.interner.kind(*ty) {
            let mut parts = Vec::new();
            collect_string_add_chain(expr, emitter.interner, &mut parts);
            if parts.len() >= 3 {
                return emit_str_build(emitter, *ty, &parts);
            }
        }
    }
    // Normal path
    let r_a = emit_expr(emitter, left);
    let r_b = emit_expr(emitter, right);
    emit_binary(emitter, *ty, op, r_a, r_b)
}
```

Where `collect_string_add_chain()` takes the whole TypedExpr (not just left/right) and recursively collects leaf nodes. And `emit_str_build()` emits each part, packs into consecutive registers, and emits StrBuild.

**Beware:** The `collect_string_add_chain` takes `&TypedExpr` and the current match arm has already destructured `expr`. Use the original `expr` reference from the outer match.

Since `emit_expr` matches on `expr` and destructures it, we need to restructure slightly. One clean approach: move the chain detection into a helper that takes the full TypedExpr and returns `Some(Vec<&TypedExpr>)` if it's a 3+ part string chain, or None:

```rust
fn try_collect_str_build_parts<'a>(
    expr: &'a TypedExpr,
    interner: &TyInterner,
) -> Option<Vec<&'a TypedExpr>> {
    if let TypedExpr::Binary { op: BinaryOp::Add, ty, .. } = expr {
        if matches!(interner.kind(*ty), TyKind::String) {
            let mut parts = Vec::new();
            collect_string_chain(expr, interner, &mut parts);
            if parts.len() >= 3 {
                return Some(parts);
            }
        }
    }
    None
}
```

Call this at the top of `emit_expr()` before the big match:
```rust
pub fn emit_expr(emitter: &mut BodyEmitter<'_>, expr: &TypedExpr) -> u16 {
    // StrBuild optimization: detect 3+ part string concatenation chains
    if let Some(parts) = try_collect_str_build_parts(expr, emitter.interner) {
        return emit_str_build(emitter, expr.ty(), &parts);
    }

    match expr {
        // ... existing match arms
    }
}
```

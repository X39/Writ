**FIX-01: Lifecycle hook dispatch**

In `dispatch.rs`, add a helper function `find_hook_by_name`:
```rust
fn find_hook_by_name(module: &writ_module::Module, type_idx: usize, name: &str) -> Option<usize> {
    let td = &module.type_defs[type_idx];
    let method_start = td.method_list.saturating_sub(1) as usize;
    let method_end = if type_idx + 1 < module.type_defs.len() {
        module.type_defs[type_idx + 1].method_list.saturating_sub(1) as usize
    } else {
        module.method_defs.len()
    };
    for idx in method_start..method_end {
        let md_name = writ_module::heap::read_string(&module.string_heap, module.method_defs[idx].name).unwrap_or("");
        if md_name == name {
            return Some(idx);
        }
    }
    None
}
```

In `domain.rs`, in `build_dispatch_table()` at line 423 where `table.insert(DispatchKey { type_key, contract_key, slot }, target)` is called, compute `type_args_hash`:

The generic type arguments are encoded in the ImplDef's `contract` MetadataToken. For a generic specialization like `Int:Into<Float>`, the contract token points to a ContractDef with specific type args. The simplest discriminator: use `impl_def.contract.0` (the raw u32 token value) as the hash. Different specializations of the same contract will have different contract tokens because they point to different TypeRef blob entries. This means the `contract_key` already encodes the specialization IF the tokens are distinct.

However, the bug at line 377-379 shows that `resolve_contract_key_for_impl` maps both `Into<Float>` and `Into<String>` to the same ContractDef row (they share the Into contract). The type argument blob is what differs.

To fix: After computing `contract_key`, also compute a `type_args_hash` from the ImplDef's contract token. Use the raw token value as a simple discriminator:
```rust
let type_args_hash = impl_def.contract.0; // raw token includes specialization encoding
```
Or better: read the contract token's TypeRef blob from the blob heap and hash it. Check if ImplDefRow has a field pointing to the generic args. If the contract token is a ContractDef (table_id=10), type_args_hash=0 (non-generic). If the contract token is a TypeRef (table_id=3), hash the TypeRef's blob bytes.

Update ALL DispatchKey construction sites to include `type_args_hash`. Check dispatch.rs for where DispatchKey is constructed for CALL_VIRT lookups — that must also include the type_args_hash. For CALL_VIRT lookup at runtime, the instruction provides a contract token + method slot. The lookup key must include the same type_args_hash that was used during table build. This may require the CALL_VIRT instruction to carry the generic specialization info, or the runtime to resolve it from context.

**Important:** If the CALL_VIRT instruction does not carry generic type args, the lookup must match. The simplest approach: use `impl_def.contract.0` as the raw discriminator for BOTH the table build (domain.rs) AND the lookup (dispatch.rs CALL_VIRT handler). This works because the compiler emits the same contract token in CALL_VIRT that it used for the ImplDef.

**FIX-03: String dereference for ExternCall args**

Use the pre-resolution approach: in the `CallExtern` handler in `dispatch.rs` (lines 555-569), after collecting args from registers but before constructing the `HostRequest::ExternCall`, resolve any `Value::Ref` args to their string content.

The cleanest approach that avoids adding a `Value::Str(String)` variant (which would cause exhaustive match failures across the codebase): add a separate `resolved_strings: Vec<Option<String>>` parallel to `args` and include it in the `HostRequest::ExternCall`.

However, this changes the `HostRequest` enum which is also used elsewhere. The simpler approach:

**Option A (recommended):** Add a `display_args: Vec<String>` field to `HostRequest::ExternCall`. This is additive (no existing field changes). The runtime fills `display_args` by resolving each arg: `Value::Int(i) => i.to_string()`, `Value::Ref(href) => heap.read_string(href).unwrap_or("<ref>")`, etc. The host uses `display_args` for display purposes while `args` retains the raw values.

**Option B (simpler):** In the CallExtern handler, before issuing the request, create a local `Vec<Value>` where `Value::Ref` entries are replaced with `Value::Int(0)` or similar placeholder AFTER extracting the string, then format a display string separately. But this loses information.

**Option C (simplest, recommended if B is too invasive):** Thread the heap reference into CliHost. Change `CliHost::new` to accept an `Arc<Mutex<GcHeap>>` or similar. But this changes the trait boundary.

**Best approach for minimal invasiveness:** In `dispatch.rs` CallExtern handler, pre-resolve strings into the args Vec itself. Since `args` is already a fresh `Vec<Value>` (copied from registers), we can mutate it. Add a special case: for each `Value::Ref(href)` in the args, try `heap.read_string(href)` and if it succeeds, leave the Ref in args but ALSO build a parallel string. Actually, the simplest path:

1. In `HostRequest::ExternCall`, add an optional `display_args: Vec<String>` field
2. In the CallExtern handler, build display_args by resolving each arg through the heap
3. In `cli_host.rs`, use `display_args` instead of `format_value` for say() output

If changing HostRequest is too invasive, use the alternative: add a `pre_resolve_args` function in dispatch.rs that transforms the args Vec before creating the request. Replace `Value::Ref(href)` with a new `Value::Str(String)` variant that only exists for ExternCall dispatch.

Actually, check first: does `Value` already have a String variant? If not, the cleanest approach is: modify only the CallExtern handler to build `display_args: Vec<String>` as a field on `HostRequest::ExternCall`. The `HostRequest` enum is defined in `host.rs`. Add the field there. Update ALL construction sites of `HostRequest::ExternCall` (likely only one: the CallExtern handler).

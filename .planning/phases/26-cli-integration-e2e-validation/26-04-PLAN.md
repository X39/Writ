**Part A: Virtual module — assign distinct contract tokens per specialization**

In `writ-runtime/src/virtual_module.rs`, the 4 collision pairs all stem from reusing the SAME contract token for different specializations:

1. `Int:Into<Float>` and `Int:Into<String>` both use `into_contract`
2. `Float:Into<Int>` and `Float:Into<String>` both use `into_contract`
3. `String:Index<Int>` and `String:Index<Range>` both use `index_contract`
4. `Array:Index<Int>` and `Array:Index<Range>` both use `index_contract`

The fix: create **specialization-specific ContractDefs** in the virtual module. Each generic specialization gets its own ContractDef (e.g., `Into_Float`, `Into_String`, `Into_Int`, `Index_Int`, `Index_Range`). These are synthetic ContractDefs — they represent `Into<Float>`, `Into<String>`, etc. Each has one ContractMethod (the `into` or `index` method with the same slot=0).

Replace the colliding ImplDef registrations:
```rust
// OLD (colliding):
builder.add_impl_def(int_type, into_contract);
add_intrinsic_method(&mut builder, "int_into_float");
builder.add_impl_def(int_type, into_contract);
add_intrinsic_method(&mut builder, "int_into_string");

// NEW (distinct tokens):
let into_float_spec = builder.add_contract_def("Into<Float>", "writ");
builder.add_contract_method("into", &[], 0);  // slot will be assigned by assign_vtable_slots or manually set to 0
let into_string_spec = builder.add_contract_def("Into<String>", "writ");
builder.add_contract_method("into", &[], 0);

builder.add_impl_def(int_type, into_float_spec);
add_intrinsic_method(&mut builder, "int_into_float");
builder.add_impl_def(int_type, into_string_spec);
add_intrinsic_method(&mut builder, "int_into_string");
```

Apply the same pattern for ALL 4 collision pairs:
- `Into<Float>`, `Into<Int>`, `Into<String>` specialization contracts for the `Into` contract
- `Index<Int>`, `Index<Range>` specialization contracts for the `Index` contract

Keep the base `into_contract` and `index_contract` definitions (they define the generic contract shape). The specialization contracts are additional, synthetic entries representing the monomorphized specializations.

**IMPORTANT:** The `Bool:Into<String>` and `String:Into<String>` impls only have ONE specialization each (no collision), so they can continue using a single contract token. However, for consistency and correctness, they should also use specialization-specific tokens. Use `into_string_spec` for all `Into<String>` impls.

Specifically, create these specialization contracts:
- `into_float_spec` = `"Into<Float>"` — used by: Int:Into<Float>
- `into_int_spec` = `"Into<Int>"` — used by: Float:Into<Int>
- `into_string_spec` = `"Into<String>"` — used by: Int:Into<String>, Float:Into<String>, Bool:Into<String>, String:Into<String>
- `index_int_spec` = `"Index<Int>"` — used by: String:Index<Int>

For Array:Index<Int> and Array:Index<Range>, apply the same index specialization contracts.

**Step 1: Determine the contract DefId at the call site**

When the compiler emits a CALL_VIRT, it knows:
- `callee_def_id`: the DefId of the method being called (e.g., the `into` method of a contract)
- The callee is a contract method on a generic receiver

The contract DefId can be found by looking at the callee's parent. The DefMap stores the parent relationship. Check `DefEntry` in `writ-compiler/src/resolve/def_map.rs` for a `parent` field or similar. The callee method's DefId is a child of the contract's DefId.

If the DefMap provides a way to get the parent DefId (contract from contract-method), use that. Then `builder.token_for_def(contract_def_id)` gives the MetadataToken.

However, the callee_def_id at a CALL_VIRT site may point to the **impl method** (not the contract method). The impl method's parent is the impl block, not the contract. We need the contract's token.

**Simpler approach — add a method to ModuleBuilder:**

Add `pub fn contract_token_for_method_def_id(&self, def_id: DefId) -> Option<MetadataToken>` to `ModuleBuilder`. This method:
1. Gets the MethodDef token via `token_for_def(def_id)`
2. If that maps to a MethodDef row, finds which ImplDef's method range includes it
3. Returns the ImplDef's `contract_token`

Implementation:
```rust
pub fn contract_token_for_method_def_id(&self, def_id: DefId) -> Option<MetadataToken> {
    let method_token = self.token_for_def(def_id)?;
    let method_row = method_token.row_index()? as usize;  // 1-based

    // Search ImplDefs for one whose method range includes this method
    for (i, impl_def) in self.impl_defs.iter().enumerate() {
        let impl_method_start = impl_def.method_list as usize;
        let impl_method_end = if i + 1 < self.impl_defs.len() {
            self.impl_defs[i + 1].method_list as usize
        } else {
            self.method_defs.len() + 1  // past the end
        };
        if method_row >= impl_method_start && method_row < impl_method_end {
            return Some(impl_def.contract_token);
        }
    }
    None
}
```

Note: `method_list` in the compiler's ImplDefRow is set during finalize. Verify that `finalize()` has been called before this method is used. Since `emit_bodies` calls `finalize()` before `emit_all_bodies()`, this is safe.

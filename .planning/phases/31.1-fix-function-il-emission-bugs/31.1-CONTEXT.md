# Phase 31.1: Fix Function IL Emission Bugs — Context

**Gathered:** 2026-03-04
**Status:** Ready for planning
**Source:** Checkpoint review during Phase 31 execution (31-02 golden test human validation)

<domain>
## Phase Boundary

Phase 31 golden tests compiled three Writ function fixtures and blessed the output. The human review checkpoint in Plan 31-02 revealed that the compiler emits structurally wrong IL in 5 distinct ways. This phase fixes those 5 bugs and re-blesses the golden fixtures with correct IL. No new features — compiler correctness only.

</domain>

<decisions>
## Implementation Decisions

### Bug 1 — CALL_INDIRECT instead of CALL for direct function calls (BUG-07)

**Location:** `writ-compiler/src/emit/body/expr.rs` ~line 266
**What's wrong:** The compiler falls through to the `CallKind::Indirect` arm even for plain named function calls, emitting:
```
CALL_INDIRECT r1, r0, r2, 0
```
where `r0` gets typed `?` (unknown) as a bogus delegate register.

**Spec:** `CALL r_dst, method_idx:u32, r_base, argc:u16` (shape CALL, 12B) is for direct static dispatch. `CALL_INDIRECT r_dst, r_delegate, r_base, argc:u16` (shape var, 10B) is only for delegate/closure calls. The discriminator is whether the callee is a statically-known method or a runtime delegate value.

**Fix:** The `CallKind` dispatch at line ~198-227 must return `CallKind::Direct(method_idx)` for plain named function calls. The `CallKind::Indirect` arm should only be reached when the callee is a delegate (first-class function value). Once fixed, the spurious delegate register (`r0 ?`) also disappears.

### Bug 2 — Wrong method signature types (BUG-08)

**Location:** `writ-compiler/src/emit/collect.rs` ~line 928-994 (`encode_fn_sig` and related)
**What's wrong:** `is_positive(n: int) -> bool` disassembles as `(bool) -> int` — parameter and return types are swapped. The signature blob has `param_count, param_types[], return_type` in that order. Either the encoding or the disassembler reads the blob fields in wrong order.

**Spec:** Signature blob layout: `param_count:u16, param_types:TypeRef[], return_type:TypeRef`. The disassembler must read param_count fields as parameters, then one field as return type — in that order.

**Fix:** Find whether the swap is in the encoder (collect.rs) or the decoder (writ-assembler disassembler). Correct the field order so `(int) -> bool` is what appears in the disassembly.

### Bug 3 — RET references void register instead of result register (BUG-09)

**Location:** `writ-compiler/src/emit/body/mod.rs` ~line 382-388 (implicit trailing return logic) + `writ-compiler/src/emit/body/expr.rs` (expr return values)
**What's wrong:** In `add(a, b) -> int`, the body produces `ADD_I r2, r0, r1` (result in r2), but `RET r3` where `r3` is void. The `RET` should emit `r2`.

**Spec:** `RET r_src` (shape R, 4B) transfers `r_src` to the caller's destination register. The register must hold a value of the method's declared return type — emitting a void register is invalid.

**Fix:** `emit_expr` for the function body must return the register holding the computed value. The trailing-return logic that appends `RET` must use that register. Likely the same root cause as Bug 4 (if/else result register).

### Bug 4 — MOV in if/else uses void source register (BUG-10)

**Location:** `writ-compiler/src/emit/body/expr.rs` ~line 617-666 (`emit_if` function)
**What's wrong:** In `factorial`, the then-branch produces `LOAD_INT r5, 1` (value in r5) but emits `MOV r4, r6` where r6 is void — r5 is never moved into the shared result register. Similarly in the else-branch.

**Spec:** `MOV r_dst, r_src` (shape RR, 6B) copies a value. r_src must be typed and hold the branch result.

**Fix:** After `emit_expr(emitter, then_branch)` returns register `r_then`, the MOV must use `r_then` as source. The bug is likely that `emit_expr` returns the wrong (or a new void) register rather than the one holding the computed value. Both Bug 3 and Bug 4 are symptoms of the same root: `emit_expr` not returning the register that actually holds the result.

### Bug 5 — Branch offsets are always 0 (BUG-11)

**Location:** `writ-compiler/src/emit/body/labels.rs` + `writ-compiler/src/emit/body/mod.rs`
**What's wrong:** `BR_FALSE r3, 0` and `BR 0` emit offset 0. The label fixup system (`apply_fixups()`) is defined in `labels.rs` but never called in the actual emit pipeline.

**Spec:** `BR offset:i32` (shape I32, 8B) and `BR_FALSE r_cond, offset:i32` (shape RI32, 8B) use signed relative byte offsets from the start of the instruction. Offset 0 is always self-referential (infinite loop) — it is never the correct value for a forward or backward branch.

**Fix:** After all instructions for a method body are emitted, call `emitter.labels.apply_fixups(...)` to patch all placeholder offsets with correct computed relative byte values. The call is missing from `emit_all_bodies` in mod.rs.

### Re-blessing and Validation (after all fixes)

After fixing all 5 bugs:
1. Run `BLESS=1 cargo test -p writ-golden` to regenerate the three `.expected` files
2. Present the re-blessed IL output to the user for a second human-validation checkpoint
3. If approved, commit the updated `.expected` files

## Specific References

**Bug locations (from code exploration):**
- `writ-compiler/src/emit/body/expr.rs`: CallKind dispatch (~line 198-272), emit_if (~line 617-666)
- `writ-compiler/src/emit/collect.rs`: encode_fn_sig (~line 928-994)

- `writ-compiler/src/emit/body/labels.rs`: apply_fixups method (~line 74)

**Spec references:**
- CALL shape: `u16(op) u32(method_idx) u16(r_dst) u16(r_base) u16(argc)` = 12B
- CALL_INDIRECT shape: `u16(op) u16(r_dst) u16(r_delegate) u16(r_base) u16(argc)` = 10B
- RET shape: `u16(op) u16(r_src)` = 4B
- MOV shape: `u16(op) u16(r_dst) u16(r_src)` = 6B
- BR shape: `u16(op) u16(pad) i32(offset)` = 8B
- BR_FALSE shape: `u16(op) u16(r_cond) i32(offset)` = 8B
- Signature blob: `param_count:u16, param_types:TypeRef[], return_type:TypeRef`

**Current wrong output (fn_basic_call.expected):**
```
CALL_INDIRECT r1, r0, r2, 0   ← should be CALL
.reg r0 ?                      ← spurious delegate reg, goes away
```

**Current wrong output (fn_typed_params.expected):**
```
.method "is_positive" (bool) -> int  ← should be (int) -> bool
RET r3  ← r3 is void; should be RET r2
```

**Current wrong output (fn_recursion.expected):**
```
MOV r4, r6   ← r6 is void; should be MOV r4, r5
RET r15      ← r15 is void; should be RET r4
BR_FALSE r3, 0  ← offset 0; should be computed byte offset
BR 0            ← offset 0; should be computed byte offset
```
